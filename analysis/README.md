# analysis/

Analyzes the benchmark's own output (`results/<model>/<model>.csv`) — kept
separate from the harness's own `requirements.txt` since pandas/matplotlib/
jupyter are analysis-only dependencies, not something the actual benchmark
run needs.

## Setup

```bash
pip install -r analysis/requirements.txt
```

## Running it

Run the benchmark first so `results/<model>/<model>.csv` files actually
exist (e.g. `./run_full_benchmark.sh`), then:

```bash
jupyter notebook analysis/benchmark_analysis.ipynb
```

or open it in any notebook-capable editor (VS Code, PyCharm Professional,
JupyterLab) and run all cells.

## What it covers

- **Per-model section** (one per model found under `results/`): pass rate by
  surface and by difficulty tier, efficiency (turns, tool calls, tokens,
  latency), and error profile. Episodes that hit a harness/infra-level error
  (a request landing while the model was still loading, a Docker blip) are
  called out and excluded from pass-rate numbers, so a serving hiccup never
  gets counted as a task failure.
- **Aggregated section**: the core comparison the benchmark exists to make —
  code-mode (Python/JS/TS) vs `json_mcp` pass rate, per model — plus overall
  pass rate, per-surface breakdown, efficiency, error profile, recovery rate
  (how often an episode with errors still passed), and harness reliability,
  all compared across the whole fleet.
- A final summary table, also saved to `analysis/summary_by_model.csv` when
  the notebook is run.

## Regenerating the notebook

`benchmark_analysis.ipynb` is generated by `build_notebook.py` rather than
hand-edited as raw JSON — change a cell's content there and re-run:

```bash
python3 analysis/build_notebook.py
```
